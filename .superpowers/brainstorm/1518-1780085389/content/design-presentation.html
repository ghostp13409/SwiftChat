Phase 2 Design: Mesh & Gossip Engine

Architectural overview of the storage and synchronization layers

1. Storage Layer (Isar Migration)

We will migrate from Drift to Isar to handle the "document-like" nature of mesh packets efficiently. Profiles, Messages, and MeshPackets will be stored in a local Isar instance.

  • MessageEntity: id, senderId, recipientId, content, timestamp, status (Pending, Sent, Delivered), isDirect (True/False).
  • MeshPacket: messageHash, recipientId, payload (encrypted), ttl, createdAt.
2. Gossip Protocol Handshake

Two devices meeting will perform a "Summary Exchange":

  1. Send a List<Map<String, int>> containing id and timestamp for all local messages.
  2. Each peer compares the list to their local DB.
  3. Peers request missing ids from each other.
  4. Missing data is sent via standard Nearby Connections payloads.
3. Smart Sync & Background Tasks

A SyncManager will coordinate discovery:

  • Scan Cycle: Discovery starts every 3 minutes for 30 seconds.
  • Cooldown: If a sync was successful with Peer X, Peer X is put on a 15-minute cooldown to save battery.
  • Cleanup: A periodic Timer (30 mins) purges messages older than the TTL (Default 24h).
4. UI Strategy

Keep the interface simple (WhatsApp style). "Delivered" status will be triggered by a "Delivery Receipt" mesh packet sent back from the recipient when they receive a message, which is then propagated back to the sender via the same gossip mechanism.
